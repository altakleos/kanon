# Plan: v0.2 Invariant Alignment — verify warn-on-unknown, upgrade rewrites sections, POSIX-only, hygiene

## Context

Originally drafted against v0.2.0a3 on 2026-04-24. Re-audited against `main` at `5d6a2d9` (post v0.2.0a4) on 2026-04-25, then re-audited again against `main` at `de6bcd3` (post v0.2.0a5 — release-only commit, no behavior change) on 2026-04-25. All five items still apply; line numbers below reflect `de6bcd3`.

A previous version of this plan was committed (`1491b0a`) and then deleted in the v0.2.0a5 release commit (`de6bcd3`) without the work shipping. This re-issue carries that plan forward, with five tweaks raised in review (see § Review tweaks below).

This plan covers small-but-real mismatches between spec, implementation, and docs that survived 0.2.0a5. None are capability gaps; they're invariant alignments and hygiene. Bundling them keeps the diff reviewable.

Contributing evidence (verified against `main` at `de6bcd3`):

1. **Spec invariant 4 vs. implementation.** `docs/specs/aspects.md` says `kanon verify` *warns* (does not fail) when a config-named aspect is absent from the installed kit. `src/kanon/cli.py:282-285` errors out and exits 1; `tests/test_cli.py:906-…` (the `test_verify_unknown_aspect` test) pins the wrong behavior. v0.2.0a4 introduced a `warnings: list[str]` plumbing path on the verify flow already (used at `cli.py:333, 342, 346, 351, 375` for other classes), so flipping unknown-aspect to a warning fits the existing pattern.

2. **`upgrade` skips AGENTS.md re-render on version match.** `src/kanon/cli.py:226-228` early-returns when `old_version == __version__ and not was_legacy`, so a consumer who hand-edited a marker-delimited section gets no re-render on `upgrade`. Decision: always rewrite the marker sections; keep the version bump conditional. Per user direction earlier in the session.

3. **Windows support status is unclear.** `src/kanon/_atomic.py:35-40` has POSIX guards; `tests/test_atomic.py:51` skips on non-POSIX; `pyproject.toml` has no OS classifier. Decision: POSIX-only for v0.2. Make it explicit in package metadata, README, and the CLI spec. Per user direction earlier in the session.

4. **E2E test hedge of unknown origin.** `tests/test_e2e_installed.py:90-94` (in `test_installed_worktrees_aspect`) demotes `worktrees` to 0 before `verify` with the comment "depth-2 branch-hygiene marker not yet injected into AGENTS.md", contradicting the in-process `test_aspect_lifecycle` which passes `verify` at depth 2. Likely stale post-38adf21; v0.2.0a4 also rebuilt the AGENTS.md base template (now aspect-neutral via `agents-md-base.md` + per-aspect body markers), which makes a regression here even less likely.

5. **`ci/check_package_contents.py` docstring rot.** Lines 3-4 still reference `docs/specs/release-process.md` and `docs/specs/release-communication.md` (don't exist); line 11 still says `sensei/__init__.py.__version__`; lines 6-7 still mention `prompts/`, `schemas/`, `profiles/`. Body code is correct (uses `_CORE_REQUIRED_FILES` referencing `kanon/...` paths) — only the header is stale.

No new spec is needed. Every change preserves (or more faithfully realises) an existing spec invariant. The CLI-spec touch in T10b is a one-line note recording the POSIX-only platform commitment, not a new capability promise.

## Review tweaks (vs. the deleted `1491b0a` draft)

Five issues raised in review of the original plan; folded in here:

- **AC9 vs. T17 contradiction resolved.** Original AC9 demanded per-commit changelog entries while T17 batched all five into one entry. This version splits CHANGELOG additions into four separate per-commit lines (one per fix-bucket), removes AC9's "not batched" wording, and replaces it with "CHANGELOG entries appear in the same commit as their corresponding behavior change."
- **T11.5 added: assert CI matrix has no Windows runner.** Removing the `skipif(os.name != "posix")` decorator means the test will *fail* (not skip) on Windows. Plan now explicitly verifies `.github/workflows/verify.yml` and `release.yml` only run POSIX matrices.
- **T10b added: spec touch-point for POSIX-only.** A user-visible platform commitment without a spec home is awkward for a spec-first kit. One-line invariant added to `docs/specs/cli.md`.
- **T7b added: assert `enabled_at` doesn't churn.** Original T6 said "gate `_write_config` on actual change" but only the heal-markers path got a test. New T7b explicitly tests that no-op `upgrade` leaves the `enabled_at` timestamp unchanged.
- **T2 clarified: append, not replace.** The human-readable tail format becomes `OK — tier-N kanon project at <path> is valid.\n  warnings:\n    - <warning text>` (multi-line append rather than inline modification). Keeps the success line stable for screen-scraping callers; warnings are clearly delimited.

## Tasks

### Verify: warn, don't fail, on unknown aspect

- [x] T1: In `src/kanon/cli.py:282-284`, change the unknown-aspect branch from `errors.append(...)` to `warnings.append(...)`. Keep the `continue` — depth/file/marker checks for an unknown aspect are meaningless and would only generate noise.
- [x] T2: In `_emit_verify_report`, when `status == "ok"` but `warnings` is non-empty, **append** to (don't replace) the success line. Format:
    ```
    OK — tier-N kanon project at <path> is valid.
      warnings:
        - <warning 1 text>
        - <warning 2 text>
    ```
    Do not change the JSON-report shape; do not change the exit code; do not modify the success-line wording. Keep stdout JSON-first, human-readable tail second.
- [x] T3: Flip the expectation in `tests/test_cli.py::test_verify_unknown_aspect` (starts at line 906): `exit_code == 0`, parsed report has `status: "ok"`, `len(report["warnings"]) >= 1`, and the warning text contains the unknown aspect name (`"bogus"` per current fixture).
- [x] T4: Add `tests/test_cli.py::test_verify_unknown_aspect_depth_out_of_range_still_fails` — a *known* aspect with out-of-range depth must still fail with non-zero exit. (If the existing `test_verify_depth_out_of_range` already covers known-aspect+bad-depth without overlap, extend it instead of duplicating.) Guards against over-loosening T1.

### Upgrade: always re-render AGENTS.md sections

- [x] T5: In `src/kanon/cli.py:226-228`, remove the `return` from the early-exit. Keep the `click.echo(f"Already at {__version__}. ...")` but let execution fall through to the AGENTS.md merge path (currently `cli.py:234-242`) and the kit.md re-render (`cli.py:244-246`). The `_merge_agents_md` + `atomic_write_text` path is already a no-op when content matches, so a version-current and clean-marker project is still silent.
- [x] T6: Gate `_write_config` (`cli.py:248`) on actual change: only call it when `old_version != __version__` or `was_legacy`. Otherwise `enabled_at` timestamps would churn on every `upgrade` invocation. Same applies to the trailing `Upgraded ... → ...` echo at `cli.py:254` — keep that gated on a real version change.
- [x] T7: Existing `tests/test_cli.py::test_upgrade_already_current` (line 358) still asserts exit 0 and "already at" — keep its assertions but allow execution to fall through. Add a new test: `test_upgrade_heals_edited_markers` — init, manually rewrite a marker section's body to garbage, call `upgrade` (version unchanged), assert the marker content was rewritten to the kit's canonical body while user content outside markers survives.
- [x] T7b: Add `tests/test_cli.py::test_upgrade_noop_does_not_churn_enabled_at` — init at tier 1, capture `aspects.sdd.enabled_at`, sleep 1.1 seconds (exceeds ISO-second precision), call `upgrade` on a version-current project, assert the post-upgrade `enabled_at` equals the captured value byte-for-byte. Locks in T6.
