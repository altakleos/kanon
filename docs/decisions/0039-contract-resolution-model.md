# ADR-0039: Contract-resolution model

## Context

[ADR-0048](0048-kanon-as-protocol-substrate.md) committed kanon to be a protocol substrate, not a kit: the substrate publishes a contract grammar and a replay engine; consumer repos and third-party publishers compose disciplines on top. ADR-0048 named the model in the abstract but did not ratify the runtime mechanics. This ADR does.

The substrate's runtime-binding problem: aspects ship prose contracts (definitions of preflight, testing, security review, release gate, etc.); a consumer's repo has a specific shape (build files, CI workflows, conventions); the substrate must connect the two without baking ecosystem knowledge into the kernel and without violating the [`P-runtime-non-interception`](../foundations/principles/P-runtime-non-interception.md) principle (no LLM in CI; no daemon supervising the agent).

The shape of the answer is **late binding via prose-contract resolution**: prose contracts are resolved by an LLM agent on a developer machine into machine-only-owned `.kanon/resolutions.yaml` artifacts; the kernel replays cached resolutions mechanically; CI never resolves, only replays. This separates the *resolver* (the agent, with full LLM cost) from the *replayer* (the kernel, deterministic and cheap), and keeps prose-as-code authoritative — the contract is the source, the resolution is the derived cache.

[ADR-0029](0029-verification-fidelity-replay-carveout.md) established the carve-out for text-only replay against committed evidence; this ADR operates within that carve-out.

## Decision

**The substrate's runtime-binding model is contract-resolution.** Prose contracts (per-aspect; published by `kanon-`, `project-`, or `acme-` namespaces) are resolved by a consumer's LLM agent into `.kanon/resolutions.yaml`, an evidence-grounded, version-pinned cache the kernel replays mechanically. Specifically:

1. **Resolutions are machine-only-owned.** `.kanon/resolutions.yaml` is generated by an agent reading prose contracts and inspecting the consumer repo. Hand-edits are forbidden; the kernel detects hand-edits via checksum and refuses replay. Consumers review resolution diffs at PR time the same way they review any other committed executable artifact.

2. **Resolutions carry quadruple version-pinning.** Every entry pins:
   - `contract-version` — the `semantic-version:` declared in the contract's frontmatter (per `P-prose-is-code`'s discipline; bumped by publishers when the contract's behaviour changes)
   - `contract-content-SHA` — SHA-256 of the contract file's full bytes (catches editorial drift even when `semantic-version` is unchanged)
   - `resolver-model` — the model identifier the agent used at resolution time (e.g., `claude-opus-4-7-2026-04-22`)
   - `evidence-SHA` — per-cited-evidence-file SHA-256, computed at resolution time

3. **The resolver runs on developer machines only; CI replays cached resolutions.** Resolution requires LLM access; replay does not. CI executes only the cached invocations. Re-resolution fires when any of the four pins drifts (contract version bumped, contract content changed, model rolled forward, evidence file edited).

4. **Resolutions are evidence-grounded.** Every entry must cite at least one repo evidence file that the agent read to determine the realization. Cited paths must exist on disk; cited SHAs must match current SHAs at replay time. A missing path is `code: missing-evidence`. A SHA mismatch is `code: sha-mismatch`. Both are `kanon verify` findings, never silent.

5. **Replay is deterministic.** Two replays of the same `resolutions.yaml` against the same repo state produce byte-equal JSON output (excluding monotonic timestamps). This is the substrate-determinism falsification probe (P2 in [`de-opinionation.md` § Self-hosting as falsification](../foundations/de-opinionation.md)).

6. **The kernel does not invoke or supervise the LLM.** The substrate ships no resolver; the agent IS the resolver. A consumer with no LLM available has a substrate that cannot generate fresh resolutions — only replay existing ones. This is intentional (per `P-runtime-non-interception`); zero-config first-run is not a substrate goal.

## Alternatives Considered

1. **Config-driven runtime (the v0.3 kit shape).** Aspect manifests declare `config-schema:` for runtime tool commands; consumers populate via `kanon aspect set-config`; the kernel renders placeholders at preflight. **Rejected.** Privileges Python (the `_detect.py` ecosystem detection); makes substitutability honour-system-only (an `acme-testing` claiming the `test-discipline` capability cannot replicate the kit's auto-fill); requires every aspect to invent its own command vocabulary. The protocol-substrate commitment retires this model (per ADR-0048).

2. **Live-LLM resolution in CI.** Every CI run invokes an agent to resolve contracts against the repo. **Rejected.** Cost: an LLM call per CI invocation; flakiness: resolutions may differ across runs; trust: the resolver runs against unmerged content during PR builds, which would extend the trust boundary beyond `repo-write-access` (per [ADR-0036](0036-secure-defaults-config-trust-carveout.md)). Cached resolutions in `.kanon/resolutions.yaml` solve all three: amortize cost across CI runs, fix the resolution at commit time, keep the trust boundary identical to `Makefile` and `package.json` scripts.

3. **Subprocess-isolated resolver.** The kernel ships a resolver process (Python + LLM client) that executes contract resolution via subprocess; the resolver is sandboxed (limited network, restricted filesystem). **Rejected.** Adds a new dependency (an LLM SDK), creates a kernel-shipped interception surface (sandboxing the agent's tool use), and contradicts `P-runtime-non-interception`. The agent IS the resolver; the substrate does not own the agent.

4. **Resolutions hand-written by the consumer.** `.kanon/resolutions.yaml` is just a YAML config the consumer authors and edits. **Rejected.** This is `Makefile`-shaped configuration with extra ceremony. The resolution's value comes from the agent's evidence-grounded inspection (citing which `package.json` script realizes "the test command" because the agent read both the contract and the manifest); a hand-authored resolution carries no such grounding. `P-prose-is-code` asks for prose to be the source — when the resolution is hand-authored, the prose contract becomes documentation, not source.

5. **Auto-resolve at every CI run with caching by content-hash.** A hybrid where CI invokes the resolver but caches by `(contract-SHA, evidence-SHAs, model-version)` and reuses the cache when those match. **Rejected.** The cache key is correct, but pushing the resolver into CI creates the trust-boundary problem from alternative #2 (PR builds run an LLM against unmerged content). Moving the resolver to developer machines keeps the trust model identical to existing repo conventions; the cache lives in `.kanon/resolutions.yaml` as a committed artifact.

## Consequences

### Substrate-level

- **`kanon-substrate` ships a resolution-replay engine** (no resolver, no LLM client, no LLM cost in CI). Phase A authors `_resolutions.py` (~280 LOC) implementing replay, stale-detection, hand-edit detection, and the JSON report shape `kanon verify` consumes.
- **`kanon resolve` is a developer-machine CLI verb.** It reads contracts, invokes the agent (via the consumer's available LLM harness), writes `.kanon/resolutions.yaml`. Phase A authors this. The kernel does not bundle an LLM client; `kanon resolve` is a thin layer that hands off to whichever harness the consumer has installed.
- **`kanon resolutions explain` is a developer-machine CLI verb.** Reports why a contract is enabled, which evidence the resolution cited, which publisher attribution applies. Phase A authors this.
- **`kanon verify` gains a resolution-conformance check.** Per the new INV-11 in `verification-contract.md` (added in this PR): `kanon verify` exit-0 includes "no resolution's structural shape is invalid; no resolution is stale against committed evidence".

### `.kanon/resolutions.yaml` artifact

- **Format**: YAML, machine-only-owned, evidence-cited, quadruple-pinned. Schema versioned via `schema-version: 1` top-level field. Future schema bumps follow the dialect-deprecation horizon (per the cadence policy; subsequent ADRs ratify the dialect grammar).
- **Stale-detection**: SHA mismatch on any of the four pins triggers re-resolution. The kernel reports `code: stale-resolution` with the offending pin.
- **Hand-edit detection**: every entry carries a `meta-checksum:` covering the entry's own bytes; the kernel computes and compares at replay time. A mismatch is `code: hand-edit-detected` and refuses replay.

### Spec layer

- **New spec `docs/specs/resolutions.md`** carries six `INV-resolutions-*` invariants ratifying machine-only-ownership, quadruple-pinning, evidence-grounding, replay-determinism, resolver-not-in-CI, and stale-failure semantics. Authored in this PR.
- **`docs/specs/verification-contract.md` gains INV-11** (exit-zero scope boundary): `kanon verify` exit-0 means conformance to enabled aspects only — not a correctness or quality endorsement. Authored in this PR. Pre-amendment body preserved per the verification-contract spec's existing immutability discipline.

### Out of scope (deferred to subsequent Phase 0 ADRs)

- **Composition algebra** (`surface:`, `before/after:` ordering across contracts targeting the same execution surface, `replaces:` substitution): ADR-0041 (realization-shape + grammar).
- **Realization-shape schema** (per-contract frontmatter declaring allowed verbs, evidence kinds, stage keys): ADR-0041.
- **Dialect grammar versioning** (`kanon-dialect:` pin; date-stamped immutable artifacts; deprecation horizon): ADR-0041.
- **Kernel/reference runtime interface** (how `kanon-substrate` discovers `kanon-reference` aspects via Python entry-points): ADR-0040.
- **Verification scope-of-exit-zero broader wording** (the public claim `kanon verify` makes about consumer-discipline correctness): ADR-0042.
- **Distribution boundary** (`kanon-substrate` / `kanon-reference` / `kanon-kit` meta-alias; release cadence): ADR-0043.
- **Substrate self-conformance** (CI gate that builds the kernel without `kanon-reference` and runs the full test suite): ADR-0044.
- **De-opinionation transition** (the migration commit sequence; `defaults:` removal; `_detect.py` deletion): ADR-0045.

### No reverse compatibility

There are no current external consumers; the cost is zero (per ADR-0048). Future forks of v0.3.x are forker-responsibility.

## Config Impact

- **`.kanon/config.yaml`** schema bumps v3 → v4. The v4 shape is sketched here (publisher-id, recipe provenance, dialect-version pin) and fully ratified by ADR-0041 alongside dialect grammar. The kanon repo's own config is rewritten in Phase 0.5.
- **`.kanon/resolutions.yaml`** is new. `schema-version: 1` opens; future schema bumps follow dialect-deprecation horizon (ratified by ADR-0041).
