# Spec: Fidelity lock — spec ↔ artifact SHA commitment

## Intent

Commit, per spec, the content SHAs of (a) the spec itself, (b) the emitted artifacts claimed to realise it, and (c) the fixtures that verify it. The file is `.kanon/fidelity.lock` at the consumer repo root, committed to version control. A single line tells a reader "does the running system match this spec?"

## Problem

Under the specs-as-source frame (`P-specs-are-source`), the reader needs to know whether the implementation is actually faithful to the spec. Without a mechanism, drift accumulates silently — a spec change lands without the implementation following, or the implementation drifts without the spec updating. Either way, the reader's confidence in the spec as source erodes.

## Sketched invariants

1. `.kanon/fidelity.lock` is YAML keyed by spec-slug.
2. Each entry records: `spec_sha`, `artifacts: [{path, sha}]`, `fixtures: [{path, sha}]`, `last_verified: <iso-timestamp>`, `last_verified_commit: <git-sha>`.
3. `kanon verify --fidelity` checks that every spec's SHA matches the file on disk; if different, the spec has changed and the lock is stale.
4. CI refuses to merge a PR where any spec SHA changed without a matching artifact SHA change (or an explicit `spec-only: true` override per entry).
5. The lock is generated by `kanon fidelity update`, not hand-edited.

## Out of Scope in v0.1

All of it. This is a `status: deferred` spec whose problem statement and sketched invariants live here so a fresh session discovers the intent.

## Why deferred

Fidelity tracking requires the consumer to have a reliable notion of which artifacts realise which spec — which in turn requires the `verified_by:` convention (`invariant-ids.md` — also deferred). Landing fidelity-lock before invariant-IDs would require authoring a looser link contract that would then need tightening; cheaper to land both together in v0.2.
