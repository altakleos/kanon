# Spec: Fidelity Lock — spec-SHA drift detection

## Intent

Detect when a spec has changed since its last verification checkpoint, so that drift between specs and their tests does not accumulate silently. A lock file records the SHA of every accepted spec at the time it was last verified. `kanon verify` compares the lock to disk and warns when a spec has changed without a corresponding lock update.

This spec covers **Phase 1: spec-SHA tracking only**. Phase 2 (fixture-SHA and artifact-SHA tracking) is deferred until the `verified-by` spec lands and provides the invariant-to-test mapping needed for richer lock entries.

## Invariants

<!-- INV-fidelity-lock-lock-file-location-format -->
1. **Lock file location and format.** The lock file lives at `.kanon/fidelity.lock` in the consumer repo root, committed to version control. It is YAML with a top-level `lock_version` key and an `entries` mapping keyed by spec filename stem:
   ```yaml
   lock_version: 1
   entries:
     aspects:
       spec_sha: "sha256:a1b2c3d4..."
       locked_at: "2026-04-24T14:02:11+00:00"
     worktrees:
       spec_sha: "sha256:e5f6a7b8..."
       locked_at: "2026-04-24T14:02:11+00:00"
   ```

<!-- INV-fidelity-lock-sha-computation -->
2. **SHA computation.** `spec_sha` is the SHA-256 hex digest of the spec file's raw bytes on disk, prefixed with `sha256:`. No normalization (no whitespace stripping, no CRLF conversion). The entire file is hashed, including frontmatter — a `status:` change is a meaningful change.

<!-- INV-fidelity-lock-included-specs -->
3. **Included specs.** The lock tracks all specs with `status: accepted` or `status: draft` that exist under `docs/specs/`. Deferred specs are excluded. The spec template (`_template.md`) is excluded.

<!-- INV-fidelity-lock-fidelity-update-command -->
4. **`kanon fidelity update` command.** A new top-level CLI command that generates or refreshes `.kanon/fidelity.lock`. It walks `docs/specs/`, filters by status, computes SHAs, and writes the lock file atomically. Idempotent — running it twice with no spec changes produces identical output.

<!-- INV-fidelity-lock-verify-integration -->
5. **`kanon verify` integration.** At SDD depth ≥ 2 (where specs exist), `kanon verify` performs fidelity checks if `.kanon/fidelity.lock` exists:
   - For each lock entry: compare `spec_sha` to the current file's SHA. Mismatch → warning ("spec X has changed since last fidelity update").
   - For each accepted/draft spec NOT in the lock: warning ("spec X is not tracked in fidelity.lock").
   - Missing lock file at depth ≥ 2: no error, no warning (lock is opt-in in Phase 1).
   - Fidelity warnings do not cause `kanon verify` to exit non-zero in Phase 1. They appear in the `warnings` array of the JSON report.

<!-- INV-fidelity-lock-lock-generated-not-edited -->
6. **Lock file is generated, not hand-edited.** The first line of the lock file is `# Generated by kanon fidelity update — do not edit manually.` The validator does not enforce this (SHAs are the source of truth), but the comment signals intent.

<!-- INV-fidelity-lock-entries-sorted-alphabetically -->
7. **Entries are sorted alphabetically** by spec slug for stable diffs in version control.

<!-- INV-fidelity-lock-core-feature-not-aspect -->
8. **Core feature, not an aspect.** Fidelity-lock is a `kanon verify` extension, not a standalone aspect. It activates based on the presence of `.kanon/fidelity.lock`, not an aspect depth dial. It is meaningful only when specs exist (SDD depth ≥ 2).

## Rationale

**Why spec-SHA only in Phase 1.** Tracking fixture and artifact SHAs requires knowing which tests cover which spec — the mapping that `verified-by` provides. Rather than block fidelity-lock on `verified-by` (deferred to v0.3), Phase 1 ships the simpler, dependency-free version: has the spec file changed? This alone catches the most common drift pattern — a spec is amended but tests are not re-run.

**Why SHA-256 of raw bytes.** Git blob hashes include a `blob <size>\0` prefix, making them non-portable outside git. SHA-256 of raw bytes is reproducible on any filesystem without git. Hashing the entire file (including frontmatter) ensures that status changes, date updates, and frontmatter amendments are tracked — all of these are semantically meaningful.

**Why YAML, not JSON.** The lock file is committed to version control and reviewed in diffs. YAML is more readable in diffs than JSON (no trailing commas, no brace nesting). It's consistent with `.kanon/config.yaml`.

**Why opt-in (no error on missing lock).** Not every project needs fidelity tracking. Projects at SDD depth 2 that don't run `kanon fidelity update` should not be penalized. The lock file's presence is the opt-in signal.

**Why a top-level command, not an aspect subcommand.** `kanon fidelity update` parallels `kanon verify` — both are project-wide operations, not aspect-scoped. The fidelity lock spans all specs regardless of which aspect they belong to.

**Why warn, not error, in Phase 1.** The workflow for responding to a stale lock is not yet established. Erroring would block `kanon verify` for a condition the user may not know how to resolve. Warnings surface the information without blocking. Phase 2 can promote to errors once the workflow is proven.

## Out of Scope

- **Fixture-SHA and artifact-SHA tracking.** Deferred to Phase 2, blocked on `verified-by` spec. Phase 2 will add `fixtures:` and `artifacts:` entries per spec in the lock file.
- **CI enforcement.** No CI script that blocks merges on stale locks. Phase 1 is advisory (warnings only). CI enforcement lands in Phase 2.
- **`spec-only` overrides.** The mechanism for marking a spec change as "no artifact update needed" is deferred to Phase 2 when artifact tracking exists.
- **Per-invariant SHA tracking.** Tracking individual invariant changes (vs whole-file changes) is deferred. Invariant-ids anchors enable this in the future but Phase 1 doesn't use them.
- **Lock file for non-spec artifacts.** The lock tracks specs only, not ADRs, plans, or design docs.
